Q1 & Q12 — How PICASso Handles New Requirement Updates (Requirements Versioning)

Maryna demonstrated the new Requirements Versioning feature, pre-released in December 2025. The system now saves all requirement versions and displays a notification when a newer version of a scoped requirement is available in the library.

  • Applies to releases in Creation, Scoping, Review & Confirm, and Manage stages.
  • Teams can click the notification to compare what changed (description updates, new requirements added to scope, etc.) and confirm whether they want to adopt the newer version.

OWASP vs. WCR Requirements Duplication (Harish)

Harish noted significant overlap between OWASP requirements and WCR (Web Configuration Requirements), forcing teams to mark many items as "Not Applicable" or "Duplicate" — a time-consuming exercise.

  • Hussain confirmed WCR will be deprecated post-April 2026 with the introduction of new CTS/CTI requirements, which include a proper mapping for web and cloud offers.
  • WCR was introduced specifically for digital offers; if a product is not classified as a digital offer, WCR does not trigger today.
  • The new CTS framework (managed by CSAC) will replace both OWASP and WCR with appropriately filtered requirement sets.

Data Privacy Toggle and Legacy DPP Requirements (Q2)

Question: "Even when the Privacy toggle is disabled, privacy requirements still appear." Maryna and Kristina explained the dual-layer design:

  • Toggle OFF: Legacy questionnaire questions (e.g., "Does the product collect personal data?") can still trigger an older set of DPP requirements inherited from the pre-PICASso Excel-based SDL process.
  • Toggle ON: Activates the full new DPP flow — risk calculation, DPP Review tab, SDL process requirements scoping, task assignments, and dedicated reviewer roles.
  • Teams who answered legacy questions affirmatively will continue to see the old DPP requirements list even with the toggle off. This is expected behaviour inherited from the legacy process.

Questionnaire Clarity — "Change of Intended Environment" Question (Meenakshi / Jeff)

The question "Is there a change of intended environment for production operation?" is too vague for brick and platform offers where the deployment environment is unknown at the time of release.

  • Hussain proposed a conditional approach: first ask "Are you aware of the intended environment?" — if yes, then ask about changes; if no, skip or adjust the logic.
  • Kristina suggested a full questionnaire review as part of the ongoing SDL process requirements work, optimising and clarifying question wording throughout.
  • Kristina confirmed the PICASso team has already drafted a guidance document explaining all questionnaire questions with examples; it will be reviewed, finalised, and published with a link from within the tool.
  • Hussain endorsed the initiative and added it as an action item to the SDL process requirements review.

Action Management — Visibility for Users Without PICASso Access (Meenakshi)

Security advisors create actions in PICASso and assign them to people (e.g., security managers, quality team members) who may not have PICASso access. These assignees cannot see or close their actions in the tool.

  • Current workarounds available:
    • If the action is submitted to Jira, the Jira ticket status syncs back to PICASso (requires Jira configured as tracking tool for the product with proper status mapping).
    • API extraction to build external reports.
    • Tableau dashboards — Hussain confirmed Industrial Automation already uses API extraction for action management reports; Tableau is the longer-term visibility solution.
  • Meenakshi noted she cannot currently download/export action management data directly from PICASso.
  • Maryna demonstrated the Jira integration configuration (selecting Jira as the tracking tool and mapping status values) that enables automatic status synchronisation.

BDBA Integration — Upcoming Improvement (Maryna)

Maryna announced that PICASso is actively working on integrating Black Duck Binary Analysis (BDBA). This will allow teams to:

  • Import vulnerable components directly from BDBA by entering only the scan ID — no manual data entry for the bulk of fields.
  • Automatically triage vulnerabilities from the BDBA scan; only a small number of fields will require manual completion.
  • Significantly reduce the effort for Software Composition Analysis (SCA) data entry.

Kristina also noted plans to integrate other SCA tools used by teams (e.g., BDH, Healer), subject to capacity and prioritisation.
